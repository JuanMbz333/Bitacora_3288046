PROYECTO FINAL FUNDAMENTOS DE SOFWARE

1. Algoritmo

Un algoritmo es un conjunto finito de instrucciones bien definidas que, al ejecutarse, resuelve un problema o realiza una tarea específica. En ingeniería de software, los algoritmos son fundamentales para diseñar soluciones eficientes, como ordenar datos o buscar información. Por ejemplo, el algoritmo de búsqueda binaria reduce significativamente el tiempo de búsqueda en listas ordenadas.

2. ciclo de vida del sofaware

El ciclo de vida del software describe las fases por las que pasa un sistema de software, desde la concepción hasta su retiro, incluyendo requisitos, diseño, desarrollo, pruebas, implementación y mantenimiento. Modelos como la cascada o Agile guían este proceso para garantizar calidad y eficiencia

3. Programación orientada a objetos

La programación orientada a objetos (POO) es un paradigma que organiza el código en objetos que combinan datos (atributos) y comportamientos (métodos). Conceptos como encapsulación, herencia y polimorfismo permiten crear sistemas modulares y reutilizables, como en lenguajes como Java o C++.

4. Depuración

La depuración es el proceso de identificar y corregir errores (bugs) en el código de un programa. Los ingenieros utilizan herramientas como depuradores, registros (logs) y pruebas unitarias para rastrear problemas y garantizar que el software funcione correctamente.

5. Control de versiones

El control de versiones es un sistema que gestiona cambios en el código fuente a lo largo del tiempo, permitiendo colaboración, seguimiento de modificaciones y reversión de errores. Herramientas como Git son esenciales para equipos de desarrollo.

6. Pruebas unitarias

Las pruebas unitarias son evaluaciones automatizadas que verifican el comportamiento de pequeñas unidades de código, como funciones o métodos, de forma aislada. Frameworks como JUnit o PyTest ayudan a garantizar que cada componente funcione correctamente.

7. API

Una API es un conjunto de reglas que permite que diferentes aplicaciones o componentes de software se comuniquen entre sí. Por ejemplo, una API REST permite a una aplicación web solicitar datos a un servidor de forma estandarizada.

8. Refactorización

La refactorización es el proceso de reestructurar el código existente para mejorar su legibilidad, mantenibilidad o rendimiento sin alterar su funcionalidad. Es una práctica clave en desarrollo ágil para mantener la calidad del software.

9. Arquitectura de software

La arquitectura de software define la estructura y organización de un sistema, incluyendo sus componentes, relaciones y principios de diseño. Patrones como MVC o microservicios guían la creación de sistemas escalables y mantenibles.

10. Integración continúa

La integración continua (CI) es una práctica en la que los desarrolladores integran frecuentemente sus cambios de código en un repositorio central, donde se ejecutan pruebas automatizadas para detectar errores rápidamente.

11. Escalabilidad

La escalabilidad es la capacidad de un sistema para manejar un aumento en la carga de trabajo (como más usuarios o datos) sin comprometer el rendimiento. Diseños como la arquitectura de micro servicios facilitan la escalabilidad horizontal.

12. Base de datos relacional

Una base de datos relacional organiza los datos en tablas relacionadas mediante claves primarias y foráneas, utilizando SQL para consultas. Ejemplos como MySQL o PostgreSQL son comunes en aplicaciones empresariales.

13. Patrones de diseño

Los patrones de diseño son soluciones reutilizables a problemas comunes en el diseño de software, como Singleton, Factory o Observer. Estos patrones mejoran la modularidad y flexibilidad del código.

14. DevOps

DevOps es una cultura y conjunto de prácticas que combinan desarrollo (Dev) y operaciones (Ops) para mejorar la colaboración, automatización y entrega continua de software de alta calidad.

15. Interfaz de usuario

La interfaz de usuario es el medio a través del cual un usuario interactúa con una aplicación, incluyendo elementos visuales como botones, formularios y menús. Un buen diseño de UI mejora la usabilidad y experiencia del usuario.

16. Experiencia de usuario

La experiencia de usuario abarca cómo un usuario percibe y siente una aplicación, incluyendo facilidad de uso, accesibilidad y satisfacción. La UX combina diseño, psicología y tecnología para optimizar la interacción.

17. Seguridad en software

La seguridad en software implica proteger sistemas contra accesos no autorizados, ataques y pérdida de datos, utilizando técnicas como cifrado, autenticación y pruebas de penetración.

18. Computación en la nube

La computación en la nube proporciona recursos computacionales (como servidores, almacenamiento y bases de datos) a través de internet, permitiendo escalabilidad y flexibilidad.

19. Testing de integración

El testing de integración verifica que diferentes módulos o componentes de un sistema funcionen correctamente juntos. A diferencia de las pruebas unitarias, se centra en las interacciones entre partes del software.

20. Documentación de software

La documentación de software incluye manuales, guías y comentarios en el código que explican cómo funciona, se usa o se mantiene un sistema. Es esencial para la colaboración y el mantenimiento a largo plazo.

21. Metodología Agile

La metodología Agile es un enfoque iterativo e incremental para el desarrollo de software que prioriza la colaboración con el cliente, la entrega continua de software funcional y la adaptabilidad a cambios en los requisitos. Frameworks como Scrum o Kanban son ejemplos comunes.

22. Backlog

Un backlog es una lista priorizada de tareas, características o requisitos pendientes de implementar en un proyecto de software, comúnmente utilizada en metodologías Agile como Scrum. El product backlog es gestionado por el Product Owner para alinear el desarrollo con los objetivos del proyecto.

23. Inyección de dependencias

La inyección de dependencias es una técnica de diseño que permite a un objeto recibir sus dependencias desde una fuente externa en lugar de crearlas internamente, promoviendo la modularidad y facilitando las pruebas unitarias.

24. Refactorización de código

La refactorización de código implica mejorar la estructura interna del código sin cambiar su comportamiento externo, con el objetivo de hacerlo más legible, mantenible y eficiente. Ejemplos incluyen renombrar variables o dividir funciones largas.

25. Modelo de datos

Un modelo de datos define cómo se organizan y relacionan los datos en un sistema, especificando entidades, atributos y relaciones. Por ejemplo, un modelo entidad-relación (ER) se usa para diseñar bases de datos relacionales.

26. Despliegue continuo

El despliegue continuo es una práctica de DevOps donde los cambios en el código se implementan automáticamente en producción tras pasar pruebas automatizadas, asegurando entregas rápidas y confiables.

27. Principio DRY

El principio DRY (Don't Repeat Yourself) busca reducir la duplicación de código al promover la reutilización a través de funciones, clases o módulos, mejorando la mantenibilidad y reduciendo errores.

28. Testing de regresión

El testing de regresión verifica que las nuevas modificaciones en el software no hayan introducido errores en funcionalidades existentes, utilizando pruebas automatizadas o manuales para garantizar estabilidad.

29. Framework

Un framework es una estructura de software predefinida que proporciona una base para desarrollar aplicaciones, ofreciendo bibliotecas, herramientas y convenciones. Ejemplos incluyen Django para Python o React para JavaScript.

30. Micro servicios

La arquitectura de micro servicios organiza una aplicación como un conjunto de servicios pequeños, independientes y comunicados a través de APIs, permitiendo escalabilidad y desarrollo paralelo.

3. Linting

El linting es el proceso de analizar el código fuente para detectar errores potenciales, inconsistencias de estilo o prácticas poco seguras, utilizando herramientas como ESLint para JavaScript o Pylint para Python.

32. Herencia

La herencia es un concepto de programación orientada a objetos donde una clase hija deriva propiedades y comportamientos de una clase padre, promoviendo la reutilización y jerarquía de clases.

33. Polimorfismo

El polimorfismo permite que objetos de diferentes clases sean tratados como instancias de una clase común, utilizando interfaces o herencia para ejecutar comportamientos específicos según el tipo de objeto. 34. Pipeline CI/CD

Un pipeline CI/CD es una serie de pasos automatizados que incluyen compilación, pruebas y despliegue de software, utilizados en integración y despliegue continuo para agilizar el desarrollo.

35. Refactorización incremental

La refactorización incremental implica realizar pequeñas mejoras continuas en el código durante el desarrollo, en lugar de grandes cambios, para mantener la calidad sin interrumpir el flujo de trabajo.

36. Middleware

El middleware es un software que actúa como intermediario entre diferentes componentes de una aplicación, como un servidor web y una base de datos, facilitando la comunicación y procesamiento.

37. Testing de carga

El testing de carga evalúa el comportamiento de un sistema bajo una carga de trabajo intensa, como múltiples usuarios simultáneos, para verificar su rendimiento y escalabilidad.

38. Patrones arquitectónicos

Los patrones arquitectónicos son soluciones estandarizadas para problemas comunes en la estructura de sistemas, como capas (layered), cliente-servidor o arquitectura orientada a eventos.

39. Automatización de pruebas

La automatización de pruebas utiliza scripts y herramientas para ejecutar pruebas de software de manera repetitiva y eficiente, reduciendo el esfuerzo manual y aumentando la cobertura.

40. Principio SOLID

Los principios SOLID son un conjunto de cinco guías de diseño (Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion) que promueven un código robusto y mantenible.

41. Principio KISS

El principio KISS (Keep It Simple, Stupid) aboga por mantener el diseño y la implementación del software lo más simple posible, evitando complejidad innecesaria para facilitar la comprensión, mantenimiento y reducción de errores.

42. Scrum

Scrum es un marco de trabajo ágil que organiza el desarrollo en iteraciones cortas llamadas sprints, con roles definidos (Scrum Master, Product Owner, equipo de desarrollo) y ceremonias como reuniones diarias y retrospectivas para mejorar la colaboración.

43. Código limpio

El código limpio se refiere a escribir código legible, bien estructurado y fácil de mantener, siguiendo prácticas como nombres descriptivos, funciones cortas y comentarios claros, como propone Robert C. Martin en su libro Clean Code.

44. Testing de aceptación

El testing de aceptación verifica que el software cumple con los requisitos del cliente o usuario final, generalmente realizado al final del desarrollo para validar que satisface las expectativas funcionales y no funcionales.

45. Reusabilidad

La reusabilidad es la capacidad de usar componentes de software (como clases, módulos o bibliotecas) en diferentes proyectos o contextos, reduciendo el tiempo de desarrollo y mejorando la consistencia.

46. Monolito

Una arquitectura monolítica es un enfoque donde todos los componentes de una aplicación (interfaz, lógica de negocio, datos) están integrados en un solo sistema, contrastando con microservicios, pero puede ser más difícil de escalar.

47. Pair programming

La programación en pareja es una técnica ágil donde dos desarrolladores trabajan juntos en un mismo código, uno escribiendo (conductor) y otro revisando (observador), mejorando la calidad y fomentando el aprendizaje.

48. ORM

Un ORM es una herramienta que facilita la interacción entre aplicaciones orientadas a objetos y bases de datos relacionales, mapeando objetos a tablas. Ejemplos incluyen Hibérnate para Java o Django ORM para Python.

49. Principio YAGNI

El principio YAGNI (You Aren’t Gonna Need It) sugiere evitar implementar funcionalidades que no son necesarias en el momento actual, reduciendo la complejidad y el esfuerzo en el desarrollo.

50. TDD

El desarrollo dirigido por pruebas es una práctica donde se escriben pruebas unitarias antes del código de producción, siguiendo un ciclo de escribir prueba, implementarla y refactorizar, para garantizar calidad y diseño robusto.

51. API Gateway

Un API Gateway es un componente que actúa como punto de entrada para gestionar y enrutar solicitudes a múltiples micro servicios, proporcionando autenticación, balanceo de carga y monitoreo.

52. Escalabilidad vertical

La escalabilidad vertical implica aumentar los recursos de un solo servidor (como CPU o RAM) para manejar mayor carga, en contraste con la escalabilidad horizontal que añade más servidores.

53. Refactorización de base de datos

La refactorización de base de datos implica modificar su estructura (tablas, índices, relaciones) para mejorar rendimiento o mantenibilidad sin alterar los datos o la lógica de la aplicación.

54. Principio de responsabilidad única

El principio de responsabilidad única (parte de SOLID) establece que una clase o módulo debe tener una sola razón para cambiar, es decir, una única responsabilidad, para mejorar la cohesión y reducir el acoplamiento.

55. Kanban

Kanban es un método ágil que utiliza un tablero visual para gestionar el flujo de trabajo, limitando el trabajo en curso (WIP) y optimizando la entrega continua mediante la visualización de tareas.

56. Logging

El logging es la práctica de registrar eventos o actividades en un sistema de software, como errores o transacciones, para facilitar la depuración, monitoreo y auditoría.

57. Inyección SQL

La inyección SQL es un tipo de ataque donde un atacante inserta código SQL malicioso en una consulta para manipular o acceder a una base de datos. Se previene usando consultas parametrizadas.

58. Contenedores

Los contenedores son unidades ligeras de software que empaquetan una aplicación y sus dependencias para ejecutarse de manera consistente en diferentes entornos, usando tecnologías como Docker.

59. Principio de abierto/cerrado

El principio de abierto/cerrado (parte de SOLID) establece que las entidades de software deben estar abiertas a extensión pero cerradas a modificación, promoviendo diseños flexibles mediante interfaces o herencia.

60. Code review

La revisión de código es el proceso de examinar el código de un desarrollador por parte de otros para detectar errores, mejorar la calidad y garantizar el cumplimiento de estándares.

61. Versionado semántico

El versionado semántico (SemVer) es un sistema para asignar números de versión a software (por ejemplo, 1.0.0) con tres partes: mayor (cambios incompatibles), menores (nuevas funcionalidades compatibles) y parche (correcciones de errores), facilitando la gestión de dependencias.

62. Testing de usabilidad

El testing de usabilidad evalúa cómo de fácil e intuitivo es usar una interfaz de usuario, recopilando retroalimentación de usuarios reales para identificar problemas y mejorar la experiencia.

63. Acoplamiento

El acoplamiento mide el grado de dependencia entre módulos de software. Un bajo acoplamiento, donde los módulos son independientes, mejora la mantenibilidad y flexibilidad del sistema.

64. Cohesión

La cohesión mide cuánto están relacionados los elementos dentro de un módulo. Alta cohesión, donde un módulo tiene una función clara y específica, es deseable para un diseño modular.

65. Refactorización de deuda técnica

La refactorización de deuda técnica aborda problemas acumulados en el código (como diseño pobre o soluciones temporales) para mejorar su calidad, a menudo priorizando tareas críticas para reducir riesgos.

66. Mocking

El mocking es una técnica de pruebas donde se crean objetos simulados para imitar el comportamiento de dependencias reales, permitiendo probar componentes de manera aislada.

67. Balanceo de carga

El balanceo de carga distribuye el tráfico de red entre múltiples servidores para optimizar el rendimiento, evitar sobrecargas y garantizar alta disponibilidad.

68. Principio de Liskov

El principio de Liskov (parte de SOLID) establece que los objetos de una subclase deben ser sustituibles por objetos de su superclase sin alterar el comportamiento del programa.

69. Continuous Delivery

La entrega continua extiende la integración continua al permitir que el software esté listo para implementarse en producción en cualquier momento, automatizando el proceso hasta el despliegue.

70. Cache

El cache es una técnica que almacena datos frecuentemente usados en una memoria rápida para reducir el tiempo de acceso, mejorando el rendimiento de aplicaciones como bases de datos o sitios web.

71. Refactorización de nombres

La refactorización de nombres implica cambiar nombres de variables, funciones o clases para que sean más descriptivos, mejorando la legibilidad y comprensión del código.

72. Principio de segregación de interfaces

El principio de segregación de interfaces (parte de SOLID) indica que los clientes no deben verse forzados a depender de interfaces que no usan, promoviendo interfaces específicas y pequeñas.

73. Código legacy

El código legacy es software antiguo que sigue en uso, pero que es difícil de mantener o actualizar debido a tecnologías obsoletas, falta de documentación o diseño desactualizado.

74. Orquestación de contenedores

La orquestación de contenedores gestiona múltiples contenedores (como en Docker) para automatizar su despliegue, escalado y comunicación, usando herramientas como Kubernetes.

75. Principio de inversión de dependencias

El principio de inversión de dependencias (parte de SOLID) sugiere que los módulos de alto nivel no deben depender de módulos de bajo nivel, sino de abstracciones, reduciendo el acoplamiento.

76. Refactorización de métodos

La refactorización de métodos implica dividir funciones largas o complejas en métodos más pequeños y específicos para mejorar la claridad y reusabilidad del código.

78. API RESTful

Una API RESTful sigue los principios de REST, utilizando métodos HTTP (GET, POST, PUT, DELETE) para interactuar con recursos identificados por URLs, promoviendo simplicidad y escalabilidad.

79. Refactorización de clases

La refactorización de clases reorganiza la estructura de clases (por ejemplo, dividiendo clases grandes o moviendo métodos) para mejorar la cohesión y reducir el acoplamiento.

80. Serverless

La arquitectura serverless permite desarrollar aplicaciones sin gestionar servidores, donde un proveedor en la nube (como AWS Lambda) ejecuta funciones en respuesta a eventos.

81. Refactorización de variables

La refactorización de variables implica mejorar la definición y uso de variables, como eliminar variables redundantes o mejorar su ámbito, para optimizar el código.

82. Testing de compatibilidad

El testing de compatibilidad verifica que el software funcione correctamente en diferentes entornos, como navegadores, sistemas operativos o dispositivos, asegurando una experiencia consistente.

83. Event-driven architecture

La arquitectura orientada a eventos organiza un sistema para responder a eventos (como clics o mensajes) mediante la publicación y suscripción a flujos de datos, ideal para sistemas reactivos.

84. Refactorización de estructuras de datos

La refactorización de estructuras de datos implica optimizar cómo se almacenan y acceden los datos en el código, como cambiar de una lista a un diccionario para mejorar el rendimiento.

85. Monitoreo

El monitoreo rastrea el rendimiento, errores y uso de un sistema en tiempo real, utilizando herramientas como Prometheus o Grafana para detectar y resolver problemas rápidamente.

86. Refactorización de bucles

La refactorización de bucles optimiza estructuras de control repetitivas, como reemplazar bucles anidados con métodos más eficientes o funciones de alto nivel, para mejorar el rendimiento.

87. Testing de seguridad

El testing de seguridad identifica vulnerabilidades en el software, como inyecciones o accesos no autorizados, utilizando técnicas como pruebas de penetración o análisis estático.

88. Refactorización de condicionales

La refactorización de condicionales simplifica estructuras lógicas complejas, como reemplazar múltiples if-else con polimorfismo o tablas de decisión, para mejorar la claridad.

89. CI/CD Pipeline

Un pipeline CI/CD automatiza la integración, prueba y despliegue de código, combinando integración continua (CI) y entrega/despliegue continuo (CD) para acelerar el desarrollo.

90. Refactorización de herencia

La refactorización de herencia ajusta jerarquías de clases para mejorar el diseño, como reemplazar herencia innecesaria con composición para reducir la complejidad.

91. Load testing

El load testing evalúa el comportamiento del sistema bajo una carga esperada, como múltiples usuarios simultáneos, para garantizar que cumple con los requisitos de rendimiento.

92. Refactorización de funciones

La refactorización de funciones implica dividir funciones largas o complejas en funciones más pequeñas y específicas para mejorar la legibilidad y reusabilidad.

93. Service mesh

Un service mesh es una capa de infraestructura que gestiona la comunicación entre microservicios, proporcionando funciones como balanceo de carga, autenticación y monitoreo.

94. Refactorización de módulos

La refactorización de módulos reorganiza el código en módulos más coherentes y con responsabilidades claras, mejorando la organización y mantenibilidad.

95. Testing de rendimiento

El testing de rendimiento mide la velocidad, estabilidad y escalabilidad de un sistema bajo diferentes condiciones, utilizando métricas como tiempo de respuesta y uso de recursos.

96. Refactorización de dependencias

La refactorización de dependencias elimina o simplifica dependencias innecesarias entre componentes, reduciendo el acoplamiento y mejorando la modularidad.

97. Domain-Driven Design

El diseño basado en dominios (DDD) se centra en modelar el software según el dominio del negocio, utilizando conceptos como entidades, agregados y contextos delimitados.

98. Refactorización de interfaces

La refactorización de interfaces ajusta las definiciones de interfaces para que sean más claras, específicas y alineadas con los principios de segregación de interfaces.

99. Blue-Green Deployment

El despliegue azul-verde utiliza dos entornos idénticos (azul y verde) para implementar nuevas versiones, alternando entre ellos para minimizar el tiempo de inactividad y facilitar reversiones.

100. Refactorización de pruebas

La refactorización de pruebas mejora la calidad de los casos de prueba, eliminando redundancias, mejorando la cobertura y haciendo que sean más fáciles de mantener.

101. Refactorización de rendimiento

La refactorización de rendimiento optimiza el código para mejorar la velocidad o el uso de recursos, como reducir la complejidad algorítmica o eliminar consultas ineficientes.

102. Testing de integración continua

El testing de integración continua ejecuta pruebas automatizadas cada vez que se integra nuevo código, asegurando que los cambios no rompan la funcionalidad existente.

103. Refactorización de documentación

La refactorización de documentación mejora la claridad y utilidad de los comentarios y guías del software, asegurando que sean precisos y fáciles de entender.

104. Circuit breaker

El patrón circuit breaker previene fallos en cascada en sistemas distribuidos, deteniendo solicitudes a un servicio que falla repetidamente hasta que se recupera.

105. Refactorización de configuración

La refactorización de configuración reorganiza archivos o parámetros de configuración para hacerlos más claros, modulares y fáciles de gestionar.

106. Testing exploratorio

El testing exploratorio es una técnica donde los probadores investigan el software sin un guion predefinido, buscando defectos mediante la exploración creativa.

107. Refactorización de excepciones

La refactorización de excepciones mejora el manejo de errores, como simplificar bloques try-catch o usar excepciones personalizadas, para hacer el código más robusto.

108. Event sourcing

El event sourcing almacena el estado de una aplicación como una secuencia de eventos, permitiendo reconstruir el estado en cualquier momento y facilitando auditorías.

109. Refactorización de concurrencia

La refactorización de concurrencia optimiza el código que maneja múltiples hilos o procesos, como mejorar la sincronización para evitar condiciones de carrera.

110. Canary deployment

El despliegue canary implementa una nueva versión del software a un subconjunto pequeño de usuarios antes de extenderlo a todos, minimizando riesgos.

111. Refactorización de acoplamiento

La refactorización de acoplamiento reduce las dependencias entre módulos o componentes de software, promoviendo un diseño más modular y flexible mediante técnicas como la inyección de dependencias o interfaces.

112. Feature toggling

El feature toggling permite activar o desactivar funcionalidades en un sistema sin modificar el código, facilitando pruebas, despliegues graduales y experimentación con nuevas características.

113. Testing de regresión automatizado

El testing de regresión automatizado utiliza scripts para verificar que los cambios recientes no hayan roto funcionalidades existentes, ejecutándose regularmente en pipelines CI/CD.

114. Refactorización de cohesión

La refactorización de cohesión mejora la relación funcional dentro de un módulo, agrupando elementos relacionados y separando los no relacionados para aumentar la claridad y mantenibilidad.

115. Idempotencia

La idempotencia asegura que una operación produzca el mismo resultado sin efectos secundarios adicionales, sin importar cuántas veces se ejecute, crucial en APIs y sistemas distribuidos.

116. Refactorización de control de flujo

La refactorización de control de flujo simplifica estructuras de control como bucles o condicionales, usando patrones más claros como guard clauses o estado en lugar de banderas.

117. Testing de humo

El testing de humo verifica las funcionalidades básicas de un sistema para asegurar que los componentes principales funcionan antes de pruebas más exhaustivas.

118. CQRS

CQRS separa las operaciones de lectura (queries) y escritura (commands) en modelos diferentes, optimizando el rendimiento y escalabilidad en sistemas complejos.

119. Refactorización de nombres de clases

La refactorización de nombres de clases renombra clases para reflejar mejor su propósito o responsabilidad, mejorando la legibilidad y alineación con el dominio.

120. Chaos engineering

La ingeniería del caos introduce fallos controlados en un sistema para probar su resiliencia y capacidad de recuperación, usando herramientas como Chaos Monkey.

121. Refactorización de parámetros

La refactorización de parámetros optimiza la lista de parámetros de funciones, reduciendo su número o agrupándolos en objetos para mejorar la claridad y mantenibilidad.

122. Testing de mutación

El testing de mutación introduce cambios deliberados (mutaciones) en el código para verificar si las pruebas existentes pueden detectar errores, evaluando la calidad del conjunto de pruebas.

123. Refactorización de herencia a composición
